# 硬件类
### ARM、AMD 和 Intel

ARM、AMD 和 Intel 都是计算机领域的重要参与者，但它们在技术架构、业务模式和应用领域等方面存在差异。具体如下：

- ARM：是一家总部位于英国的公司（2016 年被日本软银收购）。它专注于设计低功耗、高性能的处理器架构，采用精简指令集（RISC）。ARM 公司本身并不制造或出售 CPU，而是将处理器架构授权给其他厂家，通过知识产权（IP）和指令集架构授权盈利。高通、苹果、三星等公司都需要依靠 ARM 架构来生产芯片。其产品广泛应用于移动设备、嵌入式系统、物联网设备等领域，由于节能的特点，非常适用于移动通讯领域，世界上使用最广泛的 CPU 就是 ARM。
- AMD：全称 Advanced Micro Devices，是一家总部位于美国的半导体公司。主要从事处理器、图形处理器和其他计算机硬件产品的设计和制造。AMD 的处理器产品与 Intel 齐名，在个人电脑、服务器和数据中心市场上竞争激烈。AMD 以高性价比闻名于电脑界，例如其推出的 Ryzen 系列处理器，凭借出色的性能和较高的性价比受到用户和市场的好评。
- Intel：英特尔是美国一家主要从事计算机处理器研发和生产的科技公司，是芯片行业的巨头之一。Intel 的处理器采用复杂指令集（CISC），主要有酷睿（Core）、至强（Xeon）、奔腾（Pentium）、赛扬（Celeron）等系列。酷睿系列面向桌面和移动终端，至强系列主要用于服务器和工作站，奔腾和赛扬则定位低端入门级市场。Intel 在桌面端和服务器端 CPU 市场曾长期占据较大份额，具有强大的处理性能和广泛的软件兼容性，是个人计算机领域的主流处理器品牌之一。

# 软件类
### GNU libc

GNU libc，简称 glibc，是一个自由软件库，提供了大多数Linux系统上应用程序所需要基本功能。它是大多数 Linux 系统中标准的C 库，几乎所有基于Linux的程序都依赖于 glibc 来进行系统调用和提供常见的功能

**主要功能：**

1、系统调用封装：glibc 提供了与操作系统内核进行交互的接口，封装底层的系统调用，如文件操作、进程管理、内存分配

2、标准库支持：实现了 C 标准库的一些常用函数，如字符串操作、数学计算、内存分配等

3、线程支持：glibc 包含 POSIX 线程（pthread）实现，允许多线程编程

4、多语言支持：虽然 glibc 是为 C 语言设计的，但它也为其他编程语言（如C++、Fortran等）提供支持

**版本和兼容性：**

- glibc 是通过版本号来管理更新的，每个版本都会引入新特性、修复漏洞以及增加对新架构的支持。
- 较新版本的 glibc 可能会引入向后不兼容的改动，因此老的程序可能在新版本的 glibc 上无法运行。相反，较老的 glibc 可能不支持新版本的软件或硬件

## RPM（Red Hat Package Manager）

RPM 是 Linux 系统中**预编译二进制软件包管理标准**，广泛用于 RHEL、CentOS、Fedora、SUSE 等发行版，提供软件的**安装、卸载、升级、查询、校验**全生命周期管理，是系统管理与运维的核心工具之一RPM。本框架从核心概念到实战操作全面解析 RPM，适合系统管理员、运维工程师和开发人员系统掌握。

### 一、核心定义与起源

1. **全称**：Red Hat Package Manager（红帽包管理器），现更名为 RPM Package Manager（递归缩写）RPM
2. **本质**：预编译软件打包格式（`.rpm`后缀），包含二进制文件、配置文件、依赖信息、安装脚本等，可快速部署无需手动编译源码
3. **起源**：由 Red Hat 公司开发，1997 年发布 RPM 2.0，现为 Linux 标准包管理格式之一，被多数企业级发行版采用
4. **核心目标**：解决软件安装的三大痛点 ——**依赖管理、版本控制、可信任性**

### 二、RPM 包结构与命名规范

#### （一）包结构（内部组成）

一个 RPM 包本质是**包含元数据的 cpio 压缩归档文件**，结构如下：

| 组成部分              | 功能说明                                              |
| ----------------- | ------------------------------------------------- |
| **GPG 签名**        | 用于验证包的完整性和来源可信性                                   |
| **包标头（Header）**   | 元数据：名称、版本、发行号、架构、依赖、文件列表、脚本等                      |
| **有效负载（Payload）** | cpio 归档格式，包含软件的二进制文件、配置文件等                        |
| **安装 / 卸载脚本**     | % pre（安装前）、% post（安装后）、% preun（卸载前）、% postun（卸载后） |

#### （二）命名规范（标准格式）

```
name-version-release.arch.rpm
```

**各部分解析**：

- **name**：软件名称（如 nginx、mysql）
- **version**：版本号（主版本。次版本。修订号，如 1.20.1）
- **release**：发行号（包含适配系统信息，如 9.el9 表示 RHEL/CentOS 9）
- **arch**：硬件架构（x86_64、i686、noarch 等，noarch 表示跨平台）
- **后缀**：`.rpm`为二进制包，`.src.rpm`为源代码包

**示例**：`nginx-1.20.1-9.el9.x86_64.rpm`

### 三、核心特性与工作原理

1. **数据库驱动**：系统维护 RPM 数据库（默认位于`/var/lib/rpm/`），记录所有已安装包的元数据、文件信息和依赖关系，支持快速查询与校验
2. **事务机制**：安装 / 升级 / 卸载操作是原子性事务，失败时自动回滚，保障系统一致性
3. **依赖管理**：定义软件运行所需的其他包 / 库版本，安装时检查依赖关系（需手动解决或通过 YUM/DNF 自动处理）
4. **脚本支持**：提供四个阶段的自定义脚本，满足复杂安装需求
5. **文件校验**：可验证已安装文件的**大小、权限、MD5 哈希**等，检测文件是否被篡改
6. **版本控制**：精确管理软件版本，支持升级、降级和回滚操作

### 四、RPM 常用命令大全（按功能分类）

#### （一）安装操作

|命令|功能|示例|
|---|---|---|
|`rpm -ivh package.rpm`|安装包（v 显示详细信息，h 显示进度条）|`rpm -ivh nginx-1.20.1.rpm`|
|`rpm -ivh --force package.rpm`|强制安装（覆盖已安装文件）|`rpm -ivh --force nginx.rpm`|
|`rpm -ivh --nodeps package.rpm`|忽略依赖安装（慎用）|`rpm -ivh --nodeps nginx.rpm`|
|`rpm -ivh --test package.rpm`|测试安装（不实际安装，检查依赖）|`rpm -ivh --test nginx.rpm`|

#### （二）升级与降级

|命令|功能|示例|
|---|---|---|
|`rpm -Uvh package.rpm`|升级包（未安装则安装）|`rpm -Uvh nginx-1.21.0.rpm`|
|`rpm -Fvh package.rpm`|仅升级已安装的包|`rpm -Fvh nginx-1.21.0.rpm`|
|`rpm -Uvh --oldpackage package.rpm`|降级到旧版本|`rpm -Uvh --oldpackage nginx-1.19.0.rpm`|

#### （三）卸载操作

|命令|功能|示例|
|---|---|---|
|`rpm -e package_name`|卸载指定包（用包名而非文件名）|`rpm -e nginx`|
|`rpm -e --nodeps package_name`|忽略依赖卸载（慎用）|`rpm -e --nodeps nginx`|
|`rpm -e --force package_name`|强制卸载|`rpm -e --force nginx`|

#### （四）查询操作（最常用）

|命令|功能|示例|
|---|---|---|
|`rpm -qa`|查询所有已安装包|`rpm -qa \| grep nginx`|
|`rpm -qi package_name`|查询包详细信息|`rpm -qi nginx`|
|`rpm -ql package_name`|查询包安装的文件列表|`rpm -ql nginx`|
|`rpm -qf /path/to/file`|查询文件属于哪个包|`rpm -qf /usr/sbin/nginx`|
|`rpm -qd package_name`|查询包的文档文件|`rpm -qd nginx`|
|`rpm -qc package_name`|查询包的配置文件|`rpm -qc nginx`|
|`rpm -qR package_name`|查询包的依赖关系|`rpm -qR nginx`|
|`rpm -qp --scripts package.rpm`|查询包的安装脚本|`rpm -qp --scripts nginx.rpm`|

#### （五）校验与验证

|命令|功能|示例|
|---|---|---|
|`rpm -V package_name`|校验已安装包的文件完整性|`rpm -V nginx`|
|`rpm -K package.rpm`|校验包的 GPG 签名|`rpm -K nginx.rpm`|
|`rpm -Va`|校验系统中所有已安装包|`rpm -Va \| grep '^..5'`（查找 MD5 改变的文件）|

#### （六）其他实用命令

|命令|功能|示例|
|---|---|---|
|`rpm -e --test package_name`|测试卸载（不实际执行）|`rpm -e --test nginx`|
|`rpm2cpio package.rpm \| cpio -idmv`|提取 RPM 包中的文件（不安装）|`rpm2cpio nginx.rpm \| cpio -idmv`|
|`rpm --rebuilddb`|重建 RPM 数据库（数据库损坏时使用）|`rpm --rebuilddb`|

### 五、依赖管理详解

1. **依赖类型**：
    
    - 运行时依赖（ Requires ）：软件运行必需的库 / 包
    - 冲突依赖（ Conflicts ）：与其他包不兼容
    - 提供能力（ Provides ）：包提供的功能 / 库
    - 建议依赖（ Suggests ）：推荐安装的补充包
    
2. **依赖表示格式**：`package >= version`（如`libc.so.6()(64bit) >= 2.28`）
3. **依赖解决方式**：
    
    - 手动解决：查询缺失依赖→下载安装→再安装目标包
    - 自动解决：通过**YUM/DNF**（RPM 前端工具）自动下载并安装所有依赖包
    
4. **常见依赖问题与解决**：
    
    - 依赖循环：A 依赖 B，B 依赖 A→需同时安装两个包
    - 版本不匹配：依赖特定版本→指定版本安装或升级依赖包
    - 缺失依赖：找不到依赖包→检查源配置或手动下载
    

### 六、校验与安全机制

1. **文件校验**：RPM 记录每个文件的**大小、权限、所有者、组、MD5 哈希**等信息，`rpm -V`命令可检测文件是否被修改
    
    - 校验结果格式：`S M 5 D L U G T`（8 位字符，对应不同属性，`.`表示正常，`5`表示 MD5 改变）
    
2. **GPG 签名验证**：
    
    - 发行商对 RPM 包签名，确保包未被篡改且来源可信
    - 系统需导入对应 GPG 密钥（如 RHEL 官方密钥）
    - 验证命令：`rpm -K package.rpm`或`rpm --import /path/to/key.gpg`
    
3. **安全最佳实践**：
    
    - 仅从官方 / 可信源获取 RPM 包
    - 安装前验证 GPG 签名
    - 定期用`rpm -Va`检查系统文件完整性
    

### 七、RPM 与 YUM/DNF 的关系

RPM 是**底层包管理工具**，YUM/DNF 是**基于 RPM 的前端工具**，提供更友好的依赖解析和仓库管理功能：

|特性|RPM|YUM|DNF（YUM 的替代者）|
|---|---|---|---|
|**依赖解析**|手动|自动|自动（更高效）|
|**仓库支持**|不支持|支持|支持|
|**事务处理**|支持|支持（增强）|支持（更可靠）|
|**速度**|快（本地操作）|较慢|快（C++ 编写）|
|**命令复杂度**|高|低|低|
|**适用场景**|本地包管理、脚本自动化|日常系统维护、批量安装|现代系统（Fedora 22+、RHEL 8+）|

**核心区别**：RPM 专注于单个包的操作，YUM/DNF 专注于**软件仓库与依赖链管理**，自动处理复杂依赖关系

### 八、RPM 打包流程（Spec 文件开发）

RPM 打包核心是编写**Spec 文件**，定义包的构建规则与元数据：

1. **打包环境准备**：
    
    bash
    
    运行
    
    ```
    yum install rpm-build rpmdevtools
    rpmdev-setuptree  # 创建标准打包目录结构
    ```
    
2. **标准打包目录（`~/rpmbuild/`）**：
    
    plaintext
    
    ```
    ~/rpmbuild/
    ├── SPECS/        # 存放Spec文件
    ├── SOURCES/      # 存放源码包与补丁
    ├── BUILD/        # 编译目录
    ├── BUILDROOT/    # 临时安装根目录
    ├── RPMS/         # 生成的二进制RPM包
    └── SRPMS/        # 生成的源码RPM包
    ```
    
3. **Spec 文件核心结构**：
    
    spec
    
    ```
    Name:           nginx
    Version:        1.20.1
    Release:        1%{?dist}
    Summary:        High performance web server
    
    License:        BSD
    URL:            http://nginx.org/
    Source0:        http://nginx.org/download/nginx-%{version}.tar.gz
    
    Requires:       pcre-devel, openssl-devel
    
    %description
    Nginx is a high performance web server and reverse proxy.
    
    %prep
    %setup -q  # 解压源码包
    
    %build
    ./configure --prefix=/usr --with-http_ssl_module
    make %{?_smp_mflags}
    
    %install
    make install DESTDIR=%{buildroot}
    
    %files
    /usr/sbin/nginx
    /usr/share/man/man8/nginx.8.gz
    %config(noreplace) /etc/nginx/nginx.conf
    
    %changelog
    * Tue Feb 10 2026 Your Name <your.email@example.com> - 1.20.1-1
    - Initial package
    ```
    
4. **打包命令**：`rpmbuild -ba SPECS/nginx.spec`（生成二进制包与源码包）

### 九、RPM 实战与排障指南

#### （一）常见问题与解决方法

表格

| 问题现象       | 可能原因          | 解决步骤                                                                 |
| ---------- | ------------- | -------------------------------------------------------------------- |
| 安装失败：依赖缺失  | 缺少必需的库 / 包    | 1. 用`rpm -qpR package.rpm`查询依赖<br><br>2. 安装缺失依赖<br><br>3. 重新安装目标包    |
| 安装失败：文件冲突  | 与已安装包文件重叠     | 1. 用`rpm -qf /path/to/file`查询冲突文件所属包<br><br>2. 卸载冲突包或强制安装（`--force`） |
| RPM 数据库损坏  | 异常关机或磁盘错误     | 1. 备份`/var/lib/rpm/`目录<br><br>2. 执行`rpm --rebuilddb`重建数据库            |
| 校验失败：文件被修改 | 配置文件被编辑或系统被入侵 | 1. 确认修改是否合法（如配置文件）<br><br>2. 合法修改：忽略或重新打包<br><br>3. 非法修改：恢复文件或重装包    |

#### （二）最佳实践

1. **系统维护**：
    
    - 定期用`yum update`（而非`rpm -Uvh`）升级系统，自动处理依赖
    - 重要系统文件（如`/etc/passwd`）用`rpm -V`定期校验
    - 避免使用`--nodeps`和`--force`（可能导致系统不稳定）
    
2. **包管理策略**：
    
    - 生产环境使用官方源或可信第三方源
    - 自定义包使用唯一名称与版本，避免与官方包冲突
    - 记录所有手动安装的 RPM 包，便于系统恢复

## YUM 和 RPM

yum 和 rpm 都是 Linux 系统中用于软件包管理的工具，但它们的功能和使用方式有显著区别

**一、rpm（RedHat Package Manager）**

- 定义：是 顶层的软件包格式以及管理工具，最初为 Red Hat 系统开发，后来被众多 Linux 发行版采用
- 核心功能：
  - 安装本地的 .rpm 格式软件包（如 rpm -ivh package.rpm）
  - 卸载已安装的软件包（如 rpm -e package）
  - 查询软件包信息（如 rpm -qa 列出所有已安装包）
- 局限性
  - 无法自动解决依赖关系：如果安装的软件依赖其他未安装的包，rpm 会直接报错，需要手动下载并安装所有依赖
  - 只能处理本地的 .rpm 文件，不能直接从网络仓库下载安装

**二、yum（yellowdog updater modified）**

- 定义：是基于 rpm 的高级管理工具，本质上是对 rpm 的封装，解决了 rpm 的依赖问题。
- 核心功能：
  - 从配置的 软件仓库 (Repo) 中自动下载并安装软件包（如 yum install package）
  - 自动解决依赖关系的：安装软件时，会自动识别并下载所需的依赖包，无需手动干预
  - 支持批量操作（如批量安装、更新、卸载）
  - 管理软件仓库（如启用/禁用仓库、更新仓库缓存等）
- 优势：
  - 简化了软件安装流程，无需手动处理依赖
  - 可通过网络仓库获取最新软件

**三、rpm 与 yum 的关系和区别**

| 对比维度     | rpm                                  | yum                                        |
| ------------ | ------------------------------------ | ------------------------------------------ |
| 依赖处理     | 不支持，需手动解决依赖               | 自动解决依赖关系                           |
| 安装来源     | 仅本地 .rpm 文件                     | 支持本地 .rpm 和网络仓库                   |
| 操作复杂度   | 较高，需手动处理细节                 | 简单，自动化程度高                         |
| 功能定位     | 底层工具，负责 .rpm 包的基础操作     | 上层工具，基于 rpm 提供更便捷的管理能力    |
| 典型使用场景 | 安装本地单个 .rpm 包（无复杂依赖时） | 从网络仓库安装软件、批量管理软件、解决依赖 |

## EPEL Repo
### 一、核心定义与定位

 1. **CentOS base 源（基础源）**

- **全称**：CentOS Base Repository，CentOS 系统默认自带的核心软件仓库
- **维护方**：CentOS 官方团队（CentOS 项目），基于 RHEL 官方源代码构建CentOS
- **定位**：提供**操作系统核心功能**的基础软件包，是系统正常运行的基础CentOS
- **核心配置文件**：`/etc/yum.repos.d/CentOS-Base.repo`（CentOS 7/8）或分为`BaseOS.repo`和`AppStream.repo`（CentOS Stream 8+）CentOS

1. **EPEL repo（企业版 Linux 附加软件包）**

- **全称**：Extra Packages for Enterprise Linux，Fedora 社区维护的第三方附加软件仓库
- **维护方**：Fedora 特别兴趣小组（SIG），由社区开发者创建和维护
- **定位**：提供**官方 base 源中没有的高质量附加软件包**，是对 base 源的补充而非替代
- **核心配置文件**：安装`epel-release`包后生成的`/etc/yum.repos.d/epel.repo`和`epel-testing.repo`

---

### 二、核心区别对比表

表格

|特性|CentOS base 源|EPEL repo|
|---|---|---|
|**维护主体**|CentOS 官方团队（商业 / 企业级支持背景）|Fedora 社区（开源社区驱动）|
|**软件包内容**|核心系统组件（内核、glibc、systemd 等）、基础工具（yum、bash 等）|额外应用程序（如 nginx、htop、python3-pip、docker-compose 等）|
|**软件包数量**|相对较少（仅包含核心必需包）|数量庞大（提供上万种附加包）|
|**兼容性保证**|严格遵循 RHEL 兼容性标准，系统稳定性优先|设计为不与 base 源冲突或替换 base 包，专注兼容性|
|**支持周期**|与 CentOS 系统版本生命周期一致（如 CentOS 7 支持至 2024 年）|与对应 CentOS/RHEL 主版本同步|
|**官方支持**|获得 CentOS/RHEL 官方技术支持|社区支持，**红帽不提供官方支持**|
|**默认状态**|默认启用|默认不启用，需手动安装配置|
|**安全更新**|由 CentOS 官方提供及时安全补丁|由社区提供，更新速度较快|

### 三、使用场景与价值

 1. **CentOS base 源适用场景**

- 系统初始化安装与基础配置
- 安装操作系统核心组件和依赖库
- 生产环境中对稳定性和官方支持要求高的场景
- 系统安全更新和版本升级

1. **EPEL repo 适用场景**

- 需要安装官方源中没有的软件包（如现代开发工具、监控软件、第三方应用）
- 开发 / 测试环境需要更多软件选择
- 希望使用更新版本的软件（base 源软件版本通常较稳定保守）
- 部署开源软件栈（如 LNMP、Docker、Kubernetes 等）时的依赖补充

### 四、实际操作指南

1. **查看当前启用的仓库**

```
# 查看所有已启用仓库
yum repolist enabled

# 查看详细仓库信息
yum repolist all
```

2. **安装与启用 EPEL repo（CentOS 7/8）**

```
# CentOS 7/8 直接安装官方提供的epel-release包（位于CentOS Extras源）
sudo yum install -y epel-release

# 验证安装结果
yum repolist enabled | grep epel

# CentOS 8 Stream/RHEL 8 可能需要先启用CRB仓库（可选依赖）
sudo dnf config-manager --set-enabled crb
sudo dnf install -y epel-release epel-next-release
```

3. **配置国内镜像源（提升下载速度）**

方法 1：手动替换 base 源为阿里云镜像（CentOS 7）

```
sudo mv /etc/yum.repos.d/CentOS-Base.repo /etc/yum.repos.d/CentOS-Base.repo.bak
sudo curl -o /etc/yum.repos.d/CentOS-Base.repo https://mirrors.aliyun.com/repo/Centos-7.repo
sudo yum clean all && yum makecache
```

方法 2：EPEL 源使用国内镜像

编辑`/etc/yum.repos.d/epel.repo`，将`baseurl`改为国内镜像地址（如阿里云）：

```
[epel]
name=Extra Packages for Enterprise Linux 7 - $basearch
baseurl=https://mirrors.aliyun.com/epel/7/$basearch
#mirrorlist=https://mirrors.fedoraproject.org/metalink?repo=epel-7&arch=$basearch
failovermethod=priority
enabled=1
gpgcheck=1
gpgkey=file:///etc/pki/rpm-gpg/RPM-GPG-KEY-EPEL-7
```

4. **安全使用建议**
5. **避免混合第三方源**：EPEL 设计为不与 base 源冲突，但混合其他第三方源（如 Remi、RPMForge）可能导致依赖问题
6. **优先使用 base 源包**：系统核心组件尽量从 base 源安装，仅在必要时使用 EPEL 包
7. **启用 GPG 验证**：确保`gpgcheck=1`，防止安装被篡改的软件包
8. **测试环境先行**：生产环境使用 EPEL 包前，先在测试环境验证兼容性
---

### 五、总结与最佳实践

1. **基础原则**：CentOS base 源是**系统基石**，提供核心稳定性；EPEL repo 是**功能扩展**，提供额外软件选择，二者互补使用
2. **最佳实践**：
    
    - 生产环境：base 源为主，EPEL 为辅，仅在必要时使用
    - 开发环境：可同时启用 base 和 EPEL，获得更丰富的软件生态
    - 系统更新：优先使用 base 源的安全更新，EPEL 包更新需谨慎评估
    
3. **关键提醒**：EPEL 包由社区支持，**不提供红帽官方技术支持**，在严格合规的企业环境中使用前需评估风险

# 资源服务类
### ECS 

云服务器 ECS（Elastic Compute Service）是一种基于云计算技术的弹性计算服务，简单来说，它就像是一台可以在互联网上随时调用的 “虚拟电脑”，但相比传统物理服务器，它具有更强的灵活性和扩展性。

具体来说，ECS 的核心特点和作用可以从以下几个方面理解：

- 弹性伸缩：用户可以根据自己的需求（比如网站访问量、业务数据处理量）随时调整服务器的配置，比如增加或减少 CPU、内存、存储空间等，无需像传统服务器那样购买新的硬件设备。
- 按需使用：采用 “按需付费” 或 “包年包月” 等灵活的计费方式，用户只用为自己实际使用的资源付费，避免了资源浪费。
- 易管理：通过云服务商提供的控制台（如网页界面、手机 APP），用户可以远程对这台 “虚拟电脑” 进行操作，比如启动、停止、重启、安装操作系统、部署应用程序等，无需亲自到机房维护硬件。
- 高可靠性：云服务商通常会将 ECS 的资源分布在多个数据中心，即使某个硬件出现故障，服务也能快速切换到其他节点，保证业务的稳定运行，同时还提供数据备份、快照等功能，降低数据丢失风险。
- 快速部署：传统服务器需要采购硬件、安装系统和软件，可能需要几天时间，而 ECS 可以在几分钟内完成创建和配置，让用户快速上线业务（比如搭建网站、运行小程序后端、处理数据分析任务等）。 

目前，主流的云服务商（如阿里云、腾讯云、华为云、亚马逊 AWS 等）都提供 ECS 服务，它是云计算领域最基础、最常用的服务之一，广泛应用于各类互联网业务、企业信息化系统、开发测试环境等场景。

### NAS 

**1.NAS 定义**

NAS（Network Attached Storage，**网络附加存储**）是一种**基于 TCP/IP 网络**的**文件级分布式存储设备**，简单说就是一台专门提供**文件共享服务**的网络服务器，通过局域网 / 广域网为多台服务器、终端提供文件的读写、存储、共享能力，无需和服务器直连硬盘（区别于本地硬盘）。

你可以把 NAS 理解为**网络上的 “共享硬盘 / 文件服务器”**，ClickHouse 集群、Hadoop 集群等多节点架构，可通过 NAS 实现跨节点的文件共享，这也是 Broker Load 能从 NAS 导入数据的基础。

**2. NAS 核心特性（贴合大数据导入场景）**

- **文件级存储**：以**文件 / 文件夹**为存储单位（如 CSV/Parquet 数据文件、配置文件），支持主流文件协议：**NFS（Linux 环境主流）**、CIFS/SMB（Windows 环境），ClickHouse 节点通过 NFS 挂载 NAS 即可像访问本地文件一样读写数据。
- **多节点共享**：同一 NAS 目录可同时挂载到 ClickHouse 集群所有节点，所有节点访问的是**同一份文件**，避免数据在各节点重复存储，适合批量数据的统一存放。
- **即插即用 + 集中管理**：通过网络配置即可挂载使用，支持容量动态扩容（无需停机），可集中管理文件的权限、备份、访问策略。
- **低成本 + 易维护**：相比 SAN 存储（块级，高成本、复杂配置），NAS 部署简单、硬件成本低，是大数据场景中**批量数据临时存储、跨节点文件共享**的常用方案。

**3. 快速区分：NAS / DAS / SAN（避免混淆）**

大数据场景中常接触这三种存储，核心差异在**存储粒度**和**访问方式**，结合 ClickHouse 场景对比：

| 存储类型 |     全称     |        存储粒度         |          访问方式          |                    与 ClickHouse 结合场景                    |
| :------: | :----------: | :---------------------: | :------------------------: | :----------------------------------------------------------: |
| **NAS**  | 网络附加存储 | 文件级（文件 / 文件夹） |  TCP/IP 网络（NFS/CIFS）   | 存放待导入的批量数据文件（CSV/Parquet），Broker Load 统一读取 |
|   DAS    |  直连式存储  |      块级 / 文件级      | 服务器直连硬盘（SATA/SAS） |           ClickHouse 节点本地磁盘（存储分片数据）            |
|   SAN    | 存储区域网络 |    块级（磁盘分区）     |      光纤通道 /iSCSI       | 高端企业级存储，适合对性能、可靠性要求极高的核心数据存储（极少用于 ClickHouse 批量导入） |

# 其他
### .asc

类型：GPG 签名文件（`.asc` 是 OpenPGP 签名格式）。

- 用途：用于验证核心安装包（`.tgz`）的真实性和完整性，确保文件是官方发布且未被篡改。
- 验证方法：
  使用 `gpg` 工具验证签名（需先导入 Spark 官方 GPG 公钥）

---

### .sha512

- 类型：SHA-512 哈希校验文件（文本文件）。
- 内容：存储核心安装包（`.tgz`）的 SHA-512 哈希值（一串 64 字符的十六进制字符串）。
- 用途：验证下载的安装包是否完整未损坏（如下载过程中网络中断可能导致文件不完整）。
- 验证方法：
  计算本地文件的 SHA-512 哈希值，与文件中的值对比：

```
# 计算本地文件的哈希值
sha512sum spark-2.4.7-bin-hadoop2.7.tgz


# 查看校验文件中的哈希值
cat spark-2.4.7-bin-hadoop2.7.tgz.sha512
```
