# 分布式协议

## BASE理论

BASE 理论与传统 ACID（原子性、一致性、隔离性、持久性）理论相对的分布式系统中的一个重要理论，用于处理分布式数据库和系统中的事务管理。BASE 理论主要应用于 NoSQL 数据库和大规模分布式系统，其目的是在高可用性的和扩展性之间做出平衡。

### BASE 的含义

BASE 是 Basically Available，Soft state，Eventually consistent 的缩写，具体含义如下：

1、Basically Available（基本可用）

- 系统保证在大多数情况下能够响应请求，即使是部分服务或者数据不可用，系统也能尽量保证基本的服务可用性
- 这与 ACID 中的一致性和隔离性不同，在分布式系统中，可能会牺牲一些严格的事务一致性来提高系统可用性。

2、Soft State（软状态）

- 系统状态可以在某一时刻处于不一致的状态，即使没有外部输入，状态也可能随着时间而变化。数据在多个节点上是分布式的，可能存在延迟同步。
- 在 ACID 模型中，系统始终要求状态一致，而 BASE 中的软状态则是允许状态不一致，系统的最终目标是逐渐达到一致性

3、Eventually Consistent（最终一致性）

- 最终一致性是 BASE 理论的核心，意味着系统最终会达成一致状态，但是在某些时刻可能出现不一致。换句话说，在分布式系统中，数据变化可能不会立即传播到所有副本，但会在一定时间内同步，最终所有节点的数据会一致。
- 这一点与 ACID 中的一致性原则不同，ACID  中的一致性要求系统在每个时刻保持一致的状态

### BASE理论 vs ACID

- ACID是传统关系型数据库中的事务管理理论，强调数据的一致性、原子性、隔离性、持久性。在一些应用场景下，ACID 的严格要求非常重要，尤其是金融、银行等对数据一致性要求高的领域。
- BASE 则是在分布式系统中，特别是 NoSQL 数据库中提出的概念，它牺牲了一定的一致性，换取了更高的可用性和可扩展性。在大规模分布式环境下，最终一致性更能够满足一些应用的需求，

### BASE 应用场景

BASE理论非常适用于如下场景：

- **大数据平台**：如Hadoop、Cassandra等，它们采用的是最终一致性，而不是强一致性，适合处理海量的数据。 
- **NoSQL数据库**：如MongoDB、CouchDB、Cassandra等，它们一般遵循BASE原则，更加关注可扩展性和高可用性，牺牲了一些严格一致性的保证。 
- **社交网络和内容管理平台**：对于一些如社交网络（Twitter、Facebook）这类高流量的应用，允许某些时刻的数据不一致，但最终会同步。 

### BASE 例子

以**Cassandra**数据库为例：

-  当你写入一个数据时，可能会先写入到某个节点，并且这时候其他节点的数据副本还没有更新。这时候系统处于不一致的状态，但它会通过后台同步机制确保最终一致。 
-  在**最终一致性**模式下，当你从不同的数据副本读取时，可能得到不同的值。但经过一定时间后，所有副本将会同步，确保一致性。 

### BASE 总结

BASE理论主要关注在分布式系统中的可用性、软状态和最终一致性，而在大多数情况下，它能够更好地支持大规模分布式应用的需求。虽然在某些特定场景下（如金融、银行等）可能不适用，但在海量数据存储和高可用要求的场景中，BASE模型表现得非常出色。

## CAP理论

CAP（Consistency, Availability, Partition Tolerance）是分布式系统中的三大核心特性，用来描述系统在面对网络分区时的行为。在分布式系统中，CAP 定理是一个非常重要的理论框架。

### CAP 定理

CAP 定理由计算机科学家 Eric Brewer 在 2000 年提出，指的是：

**Consistency (一致性)**：

- 所有节点看到的数据是一致的。在一个操作完成后，所有用户都看到同样的数据。即使系统中的数据被更新，所有后续请求都会看到更新后的数据。 

**Availability (可用性)**：

- 每个请求都会得到响应，不管请求是否成功。在系统中发生故障时，服务仍然可用，并且返回某种形式的结果。 

**Partition Tolerance (分区容错性)**：

- 当网络分区发生时（即网络故障或节点之间无法通信时），系统仍然能够继续运行，并尽量保持一致性和可用性。分区容错性是分布式系统中非常重要的一项能力，因为网络分区是不可避免的。 

### CAP 定理的本质

CAP 定理表明，在一个分布式系统中，最多只能同时满足以下两项特性：

- **CA**（一致性 + 可用性）：适用于单一节点的系统或者可以接受完全的主从复制。 
- **CP**（一致性 + 分区容错性）：系统在发生网络分区时依然能保持一致性，但可能牺牲部分可用性。 
- **AP**（可用性 + 分区容错性）：系统保持可用性，即使发生分区，可能会出现不同步的现象（不一致性）。 

### CAP 例子

**CA (一致性 + 可用性)**：

1.  比如一个单机数据库，所有的读写操作都是在同一台机器上进行的，数据一致并且操作是可用的。 
2.  这种情况不会发生网络分区，所以没有分区容错的需求。 

**CP (一致性 + 分区容错性)**：

1.  例如，HBase 在分布式环境下是典型的 CP 类型系统，保证一致性，在发生分区时，系统会拒绝部分请求，保证数据一致性。 
2.  一旦分区发生，系统可能会暂停可用性以确保一致性。 

**AP (可用性 + 分区容错性)**：

1.  比如 Cassandra、Couchbase 等。它们在分区的情况下，仍然会保证可用性，数据可能存在不同步的情况，但总会返回某种结果。 

### CAP 如何选择

- **一致性 vs 可用性**：如果你的应用需要强一致性，例如金融系统，通常会选择 **CP**，因为它能够确保系统内所有的数据都是同步的。 
- **可用性 vs 一致性**：如果你的应用对可用性要求很高，比如社交媒体应用（如 Twitter），可以选择 **AP**，保证系统能持续响应请求，即使数据存在暂时的不一致。 
- **分区容错性**：分区容错性是分布式系统的基本需求，因此分区容错通常是必须的，许多系统会牺牲一致性来保证可用性。 

### CAP 总结

CAP 定理告诉我们，分布式系统无法同时满足一致性、可用性和分区容错性三者，它只能满足其中的两项。因此，在设计分布式系统时，需要根据实际需求权衡这三个特性。

## ZAB 协议

ZAB（ZooKeeper Atomic Broadcast）是 ZooKeeper 专为分布式协调服务设计的崩溃恢复的一致性协议，用于保证集群中所有节点的数据一致性。它是 Paxos 协议的变种，但针对 ZooKeeper 的写请求顺序性和快速恢复进行了优化。

### ZAB 协议的核心目标

1. 消息顺序性（Total Order）
   1. 所有写请求（事务）必须按照全局唯一顺序执行（先 Leader 提案，再广播给 Follower）。
2. 崩溃恢复（Recovery）
   1. Leader 崩溃后，新 Leader 能快速恢复集群状态，确保数据不丢失。
3. 高吞吐（Throughput）
   1. 通过 Leader 单点处理写请求 + 批量广播（类似 2PC），提高性能。

### ZAB 协议的两种模式

ZAB 协议在运行时会根据集群状态切换模式：

| 模式                  | 作用                                                         |
| --------------------- | ------------------------------------------------------------ |
| 崩溃恢复（Recovery）  | 选举新 Leader，并同步集群数据到最新状态（保证一致性）。      |
| 消息广播（Broadcast） | Leader 接收写请求，按顺序广播提案（Proposal），多数派确认后提交（Commit）。 |

### ZAB 崩溃恢复模式（Leader 选举）

当集群启动或 Leader 崩溃时触发：

1. 选举新 Leader
   1. 所有节点进入 `LOOKING` 状态，通过 Fast Leader Election 算法（基于 `zxid` 和 `myid`）投票选出 Leader。
   2. `zxid`（事务ID）越大优先级越高，`myid`（节点ID）作为次要条件。
2. 数据同步
   1. 新 Leader 检查其他节点的 `zxid`，将缺失的数据同步给 Follower。
   2. 确保所有节点的数据状态一致后，切换到消息广播模式。

**关键点：**

- zxid 结构：`高32位 epoch（任期）` + `低32位 计数器`，避免旧 Leader 复活后产生脑裂。
- 多数派原则：必须得到多数节点（`n/2+1`）确认才能成为 Leader。

### ZAB 消息广播模式（事务处理）

Leader 正常工作时，处理客户端写请求的流程：

1. 接收请求
   1. Leader 为每个写请求生成一个全局唯一的 `zxid`（递增）。
2. 广播提案（Proposal）
   1. Leader 将提案（`zxid + 数据变更`）发送给所有 Follower。
3. 收集确认（ACK）
   1. Follower 收到提案后写入本地日志，并回复 ACK。
4. 提交（Commit）
   1. Leader 收到多数派 ACK 后，广播 `COMMIT` 命令，所有节点正式提交数据。

**关键点：**

- 顺序性保证：所有提案按 `zxid` 顺序处理（先到先服务）。
- 读请求优化：读操作可直接由 Follower 处理（最终一致性），但写必须走 Leader。

### ZAB 与 Paxos 的区别

| 特性        | ZAB                         | Paxos                           |
| ----------- | --------------------------- | ------------------------------- |
| 设计目标    | 高吞吐的顺序消息广播        | 通用一致性协议                  |
| Leader 角色 | 必须有固定 Leader（写单点） | 无固定 Leader（每次提案可不同） |
| 消息顺序    | 严格保证顺序（zxid 递增）   | 不保证全局顺序                  |
| 恢复速度    | 快速恢复（依赖 epoch 机制） | 恢复较复杂                      |
| 使用场景    | ZooKeeper 等协调服务        | 分布式数据库、锁服务等          |

### ZAB 如何避免脑裂

ZAB 通过以下机制防止脑裂（多个 Leader 同时存在）：

1. epoch 任期号
   1. 每次新 Leader 选举后，`epoch` 递增，旧 Leader 的提案会被拒绝。
2. zxid 全局唯一
   1. 新 Leader 的 `zxid` 一定大于旧 Leader 的，确保数据不会回退。
3. 多数派原则
   1. 只有获得多数派投票的节点才能成为 Leader。

### ZAB 的提案广播代码示例

（伪代码展示核心逻辑）

```
// Leader 处理写请求
class Leader {
    void processWriteRequest(Request request) {
        // 1. 生成 zxid
        long zxid = generateZxid(); 
        
        // 2. 广播提案
        Proposal proposal = new Proposal(zxid, request);
        for (Follower follower : followers) {
            follower.sendProposal(proposal);
        }


        // 3. 等待多数派 ACK
        waitForMajorityAcks(zxid);


        // 4. 提交
        broadcastCommit(zxid);
    }
}


// Follower 处理提案
class Follower {
    void handleProposal(Proposal proposal) {
        // 写入本地日志（未提交）
        writeToLog(proposal);
        
        // 回复 ACK
        sendAck(proposal.zxid);
    }
}
```

### ZAB 的典型应用场景

1. ZooKeeper 集群：保证配置变更、节点创建等操作的强一致性。
2. 分布式锁：通过顺序 zxid 实现公平锁。
3. 选举服务：快速选出主节点（如 Kafka Controller 选举）。

### ZAB 总结

- ZAB = 崩溃恢复 + 消息广播，是 ZooKeeper 强一致性的核心。
- 顺序性：通过 `zxid` 保证所有操作的全局顺序。
- 高可用：快速 Leader 选举 + 数据同步，实现故障恢复。
- 与 Paxos 的区别：ZAB 更注重顺序性和吞吐量，适合协调服务场景。

ZAB 的设计完美契合了 ZooKeeper 的需求，使其成为分布式系统中最常用的协调服务之一。 

## Raft 协议

 **Raft** 是一种用于分布式系统中的一致性算法，它的目标是使得在分布式系统中，即使某些节点失败，系统也能够保持一致性。Raft 算法是由 Diego Ongaro 和 John Ousterhout 在 2014 年提出的，是 **Paxos 算法** 的一种易于理解的替代方案，常用于实现分布式系统中的 **领导选举**、**日志复制** 和 **故障恢复**。

### 📌 **Raft 算法的目标**

Raft 主要目标是确保 **分布式系统的一致性**，即所有节点都能够看到相同的数据，并且在节点发生故障时，系统能继续提供服务。它通过以下几个核心概念实现这一目标：

1. **领导者选举（Leader Election）**：确保系统有一个统一的决策者，避免多个节点竞争和冲突。 
2. **日志复制（Log Replication）**：将操作记录（日志）从领导者节点同步到其他节点，确保数据一致性。 
3. **安全性（Safety）**：确保系统中的数据在节点失败后仍然是安全和一致的。 

### 📌 **Raft 算法的核心概念**

**节点状态**： Raft 中的每个节点都可以有三种状态：

1. **领导者（Leader）**：负责所有客户端请求的处理和日志的复制。 
2. **跟随者（Follower）**：被动地响应领导者的请求，不主动发起请求。 
3. **候选者（Candidate）**：当跟随者节点无法与领导者通信时，它会尝试竞选成为领导者。 

**领导者选举（Leader Election）**：

 

1.  当一个系统启动时，所有的节点都会处于 **跟随者状态**。系统中只允许有一个 **领导者**，如果领导者失效（由于网络分区、崩溃等原因），就会触发 **选举**。 
2.  在选举过程中，节点会向其他节点发送 **请求投票**，如果大多数节点投票给某个节点，该节点就成为 **领导者**。 
3.  领导者负责处理客户端请求，并将日志复制到其他节点。如果领导者崩溃或失联，系统会通过选举产生新的领导者。 

**日志复制（Log Replication）**：

 

1.  所有操作都会记录在 **日志** 中，领导者负责将操作日志复制到 **所有的跟随者节点**。 
2.  每次日志的写入操作，领导者会将日志条目发送给所有跟随者。当一个跟随者收到领导者的日志条目后，会返回一个确认消息，表示它已经将日志条目存储下来。 
3.  只有当大多数节点（包括领导者）确认日志条目已经提交时，领导者才会提交这个日志条目，即应用该操作。 

**安全性（Safety）**：

 

1. **Leader 提交**：只有当领导者确认所有的跟随者都已经保存该条日志，才会将日志条目标记为已提交。 
2. **日志一致性**：Raft 保证，只有在日志条目在多数节点中提交后，它才能被应用。 
3. **领导者更新日志**：在每个新的领导者当选时，Raft 确保领导者能够根据 **最前面的共同日志** 执行操作，保证一致性。 

**日志条目的持久化**：

 

1.  每个日志条目包含一个 **索引** 和 **任期号**，确保日志条目的唯一性和顺序性。 
2.  节点在持久化日志条目时，必须保证写入磁盘，以避免在节点崩溃时丢失数据。 

**客户端请求**：

 

1.  客户端向系统发送请求，只有当请求的日志条目被成功复制到多数节点时，领导者才会向客户端返回成功响应。 

 

### 📌 **Raft 算法的工作流程**

**领导者选举**：

1.  所有节点最开始都处于 **跟随者** 状态。 
2.  如果一个跟随者节点在一定时间内没有收到领导者的心跳，它会变成 **候选者** 状态，开始发起选举。 
3.  如果一个候选者节点获得大多数节点的投票，它就成为 **领导者**。 

**日志复制**：

1.  领导者将客户端请求转换为日志条目，并将其广播给所有跟随者。 
2.  跟随者接收到日志条目后，会存储并返回确认消息。 
3.  一旦大多数节点确认接收了该日志条目，领导者会将日志条目标记为提交，并应用它。 

**日志一致性**：

1.  每个节点都有一个日志索引，每个日志条目都有一个任期号。 
2.  领导者确保它的日志条目是经过一致性检查的，只有在大多数节点确认接收到该条日志后，它才能被标记为提交。 

**故障恢复**：

1.  如果某个节点崩溃或失联，它会在恢复后从领导者那里同步最新的日志。 
2.  领导者通过向节点发送缺失的日志条目来恢复节点的数据一致性。 

### 📌 **Raft 算法与 Paxos 的比较**

Raft 和 Paxos 都是为了解决分布式一致性问题的算法，但 **Raft** 更容易理解，并且提供了 **更清晰的领导者选举机制**。

- **Paxos** 相对较复杂，尤其是在领导者选举和一致性保障上。 
- **Raft** 引入了明确的领导者角色，并且通过 **日志复制** 和 **安全性保证** 来确保系统的一致性，使得它更容易实现和理解。 

### 📌 **Raft 在实际中的应用**

Raft 已经被广泛应用于许多分布式系统和数据库中，如：

- **Consul**：用于服务发现的分布式系统。 
- **etcd**：用于存储和管理配置信息的分布式系统，广泛用于 Kubernetes。 
- **RocksDB**：在 RocksDB 的分布式扩展中，使用了 Raft 来管理分布式日志。 

### 📌 **总结**

**Raft** 是一个为分布式系统设计的一致性算法，主要通过 **领导者选举**、**日志复制** 和 **安全性保证** 来实现一致性。它通过清晰的逻辑和较易实现的方式，解决了在分布式环境下如何保证数据一致性的问题。

- **领导者选举**：确保系统始终有一个统一的决策者。 
- **日志复制**：保证所有节点的数据一致性。 
- **故障恢复**：保证系统在节点失效后依然能恢复正常。 

Raft 的易理解性和清晰的实现方式使它成为分布式系统中广泛使用的一致性协议。



# 网络协议

## SFTP

SFTP（全称 **SSH File Transfer Protocol**，SSH 文件传输协议）是一种基于 SSH（Secure Shell）协议的安全文件传输协议，用于在客户端和服务器之间加密传输文件。它的核心特点是**所有数据（包括文件内容、用户名、密码、命令等）都通过加密通道传输**，避免了传统 FTP 协议明文传输带来的安全风险（如数据被窃听、篡改）。

### 一、SFTP 与其他协议的区别

| 协议     | 核心特点                                                     | 安全性               | 端口默认值  |
| -------- | ------------------------------------------------------------ | -------------------- | ----------- |
| **SFTP** | 基于 SSH 协议，加密传输所有数据，支持文件管理（上传、下载、删除等） | 高（全程加密）       | 22          |
| FTP      | 传统文件传输协议，明文传输数据和密码，无加密                 | 低（易被窃听）       | 21          |
| FTPS     | FTP over SSL/TLS（在 FTP 基础上添加 SSL 加密），需额外配置证书 | 中（加密但配置复杂） | 990（默认） |

**关键优势**：SFTP 无需额外配置加密层（依赖 SSH 的加密能力），且仅通过一个端口（22）即可完成所有操作，比 FTPS 更简单、更安全，是目前主流的安全文件传输方案。

### 二、SFTP 的工作原理

1. **建立 SSH 连接**：客户端先通过端口 22 与服务器建立 SSH 连接，完成身份验证（用户名 + 密码，或 SSH 密钥对）。
2. **加密通道传输**：连接建立后，所有文件传输命令（如上传、下载）和数据都通过 SSH 的加密通道传输，第三方无法解密。
3. **文件操作**：支持标准文件管理功能，如上传（put）、下载（get）、删除（rm）、创建目录（mkdir）等，操作方式类似 FTP，但全程加密。

### 三、SFTP 的典型应用场景

1. **敏感数据传输**：企业内部传输财务报表、用户数据等敏感文件，避免泄露。
2. **服务器管理**：通过 SFTP 向云服务器（如 Linux 服务器）上传代码、配置文件（替代 FTP 的不安全方式）。
3. **跨网络文件同步**：在不同设备（如本地电脑与远程服务器）之间安全同步文件。

### 四、如何使用 SFTP？

#### 1. 客户端工具（可视化操作，适合新手）

- **FileZilla**（跨平台，免费）：支持 SFTP、FTP、FTPS，界面简单，输入服务器 IP、端口（默认 22）、用户名、密码即可连接。
- **WinSCP**（Windows，免费）：专注于 SFTP 和 SCP，支持拖拽文件传输，适合 Windows 用户。
- **Termius**（跨平台）：支持 SSH 和 SFTP，手机端也可使用，适合远程管理。

#### 2. 命令行操作（适合开发者 / 服务器管理）

在 Linux/macOS 终端或 Windows 的 PowerShell 中，直接通过`sftp`命令连接服务器：

```bash
# 基本格式：sftp 用户名@服务器IP/域名
sftp user@192.168.1.100  # 连接服务器，默认端口22（若端口非默认，加 -P 端口号：sftp -P 2222 user@ip）

# 连接成功后，常用命令：
put local_file.txt  # 上传本地文件到服务器当前目录
get server_file.txt  # 下载服务器文件到本地当前目录
ls  # 查看服务器当前目录文件
cd /path  # 切换服务器目录
lcd /local/path  # 切换本地目录
rm server_file.txt  # 删除服务器文件
exit  # 退出连接
```

### 五、核心优势总结

1. **安全性高**：基于 SSH 加密，所有数据传输全程加密，防窃听、防篡改。
2. **功能全面**：除文件传输外，还支持文件权限修改、目录管理等操作。
3. **配置简单**：仅需一个端口（22），无需额外配置加密证书（依赖 SSH 密钥或密码验证）。
4. **兼容性强**：几乎所有 Linux/Unix 服务器默认支持 SFTP（因 SSH 是标配），Windows 服务器也可通过安装 OpenSSH 组件支持。

总之，SFTP 是替代传统 FTP 的首选方案，尤其适合需要传输敏感数据（如企业文件、个人隐私数据）的场景，兼顾安全性和易用性。

## Thrift

Thrift 协议是 Apache Thrift 跨语言服务框架的核心组成部分，主要负责**定义数据的序列化 / 反序列化格式**和**远程服务调用（RPC）的交互规范**，是实现不同语言程序间高效通信的基础。它并非单一协议，而是一套可扩展的协议族，支持多种编码格式，适配不同性能和可读性需求。

### 一、Thrift 协议的核心作用

Thrift 协议解决了两个关键问题：

1. **数据如何 “编码”**：定义内存中的数据结构（如结构体、列表）如何转换为可传输的字节流（序列化），以及如何从字节流还原为数据结构（反序列化）。
2. **服务如何 “调用”**：定义远程服务的接口（如方法名、参数类型、返回值类型），确保客户端和服务端对 “调用什么方法、传什么参数” 达成共识。

### 二、Thrift 协议的核心特性

1. **跨语言支持**通过统一的接口定义语言（IDL）描述数据结构和服务接口，Thrift 编译器可生成 20+ 种语言（Java、Python、C++、Go 等）的代码，确保不同语言的程序能基于相同的协议规则通信。

2. **多协议可选**Thrift 提供多种序列化协议，可根据场景（性能、可读性、带宽）灵活选择，核心协议如下：

   | 协议类型              | 编码方式                  | 特点                                                 | 典型场景                     |
   | --------------------- | ------------------------- | ---------------------------------------------------- | ---------------------------- |
   | `TBinaryProtocol`     | 二进制编码                | 结构紧凑，无冗余元数据，性能较高（默认选择）         | 生产环境（追求效率）         |
   | `TCompactProtocol`    | 紧凑二进制（变长编码）    | 比 `TBinary` 体积小 30%+，对整数、字符串采用压缩编码 | 带宽受限场景（如跨机房通信） |
   | `TJSONProtocol`       | JSON 格式                 | 可读性强，但体积大、序列化 / 反序列化效率低          | 调试或跨平台可读性需求       |
   | `TSimpleJSONProtocol` | 简化 JSON（仅值，无类型） | 比 `TJSON` 更轻量，适合脚本语言（如 Python）解析     | 轻量 JSON 交互               |
   | `TDebugProtocol`      | 人类可读文本（键值对）    | 包含详细类型和字段信息，仅用于调试                   | 开发阶段调试                 |

3. **数据类型与结构支持**Thrift 协议通过 IDL 定义数据类型，确保序列化 / 反序列化时的一致性，支持：

   - 基本类型：`bool`、`byte`、`i16`、`i32`、`i64`、`double`、`string`；
   - 容器类型：`list<T>`（有序列表）、`set<T>`（无序去重）、`map<K,V>`（键值对）；
   - 复杂类型：`struct`（结构体，类似类）、`enum`（枚举）、`exception`（异常）、`service`（服务接口）。

   示例 IDL 定义（体现协议需处理的结构）：

   ```thrift
   struct Order {
     1: required i64 id,          // 必选字段（序号用于序列化）
     2: optional string product,  // 可选字段
     3: map<string, string> attrs // 容器类型
   }
   service OrderService {
     Order getOrder(1: i64 orderId) // 服务接口（协议需定义调用规则）
   }
   ```

### 三、Thrift 协议的工作流程（以 RPC 为例）

1. **定义 IDL**：用 IDL 描述数据结构（如 `Order`）和服务接口（如 `OrderService`），作为客户端和服务端的 “契约”。
2. 生成代码：通过 Thrift 编译器（thrift --gen <语言> order.thrift）生成目标语言的代码，包含：
   - 数据结构的序列化 / 反序列化逻辑（基于所选协议）；
   - 服务接口的客户端代理（负责按协议编码请求）和服务端骨架（负责按协议解码请求）。
3. 客户端调用：
   - 客户端创建数据对象（如 `Order`），调用生成的代理方法（如 `getOrder(123)`）；
   - 代理按所选协议（如 `TBinaryProtocol`）将方法名、参数序列化为字节流，通过传输层（如 `TSocket`）发送给服务端。
4. 服务端处理：
   - 服务端通过传输层接收字节流，按协议反序列化为方法名和参数；
   - 调用服务实现类的对应方法（如 `getOrder`），处理后将结果序列化并返回。
5. **客户端解析**：客户端接收响应字节流，反序列化为结果对象（如 `Order`），完成调用。

### 四、Thrift 协议 vs 其他主流协议

| 协议     | 特点对比                                                     |
| -------- | ------------------------------------------------------------ |
| Thrift   | 支持多协议（二进制 / JSON 等），跨语言能力强，性能优异，适合内部服务 RPC |
| Protobuf | 仅二进制协议，序列化效率略高于 Thrift，语法更简洁，但无内置 RPC 框架支持 |
| JSON     | 文本格式，可读性强，跨平台通用，但体积大、效率低，适合人机交互或弱性能场景 |
| gRPC     | 基于 HTTP/2 和 Protobuf，支持流式通信，适合跨网络服务，但语言支持不如 Thrift 全 |

### 五、总结

Thrift 协议是一套**跨语言、可扩展的序列化与 RPC 规范**，通过 IDL 统一接口定义，结合多种编码格式（二进制 / JSON 等），实现了不同语言程序间的高效通信。它的核心优势在于 “灵活性”（多协议可选）和 “兼容性”（广泛语言支持），因此被广泛用于分布式系统中（如 Hive、HBase、Cassandra 等）的服务间通信，尤其适合对性能和跨语言交互有较高要求的场景。



# 标准

## ANSI SQL标准

在技术领域（尤其数据库、编程、标准规范场景），**ANSI** 核心指 **American National Standards Institute（美国国家标准协会）**—— 一个非盈利性的标准制定组织，负责制定和推广各类行业技术标准，确保不同厂商的产品 / 技术之间具备兼容性、互操作性。

对开发者（尤其是你关注的 SQL、大数据领域）而言，**ANSI 最常用的场景是 “ANSI SQL 标准”**，这也是与你工作最相关的部分，以下展开说明：

### 一、核心场景：ANSI SQL 标准（数据库领域）

ANSI 联合国际标准化组织（ISO）制定了 **SQL 语言的统一标准（ANSI/ISO SQL）**，目的是：

- 统一不同数据库（MySQL、Oracle、StarRocks、Presto、PostgreSQL）的 SQL 语法规则；
- 让开发者编写的 SQL 语句能 “跨数据库兼容”，避免因厂商自定义语法导致的 “一套 SQL 只能在一个数据库中运行”。

#### 1. ANSI SQL 包含的核心标准内容

所有主流数据库都遵循的基础语法（即 ANSI 标准部分）：

- 基础查询：`SELECT、FROM、WHERE、GROUP BY、HAVING、ORDER BY`；
- 多表关联：`INNER JOIN、LEFT JOIN、RIGHT JOIN`（标准 JOIN 语法，区别于 MySQL 早期的逗号分隔关联）；
- 数据操作：`INSERT、UPDATE、DELETE`；
- 事务控制：`COMMIT、ROLLBACK、SET TRANSACTION`（ACID 特性的标准定义）；
- 常用函数：`COUNT()、SUM()、AVG()、MAX()、MIN()` 等聚合函数，`SUBSTRING()、DATE()、CASE WHEN` 等函数。

#### 2. 标准 vs 厂商扩展语法

ANSI SQL 是 “基础通用语法”，各数据库厂商会在标准之上增加**自定义扩展语法**（非 ANSI 标准），例如：

| 功能场景     | ANSI 标准语法                                   | 厂商扩展语法（非 ANSI）                                      |
| ------------ | ----------------------------------------------- | ------------------------------------------------------------ |
| 分页查询     | 无统一标准（ANSI SQL:2008 新增 `OFFSET/FETCH`） | MySQL 的 `LIMIT`、Oracle 的 `ROWNUM`                         |
| 日期函数     | `DATEADD(day, 1, date)`                         | MySQL 的 `DATE_ADD(date, INTERVAL 1 DAY)`、Oracle 的 `date + 1` |
| 关联查询简写 | 必须显式写 `JOIN...ON`                          | MySQL 支持 `FROM t1, t2 WHERE t1.id = t2.id`（旧式语法，非 ANSI 推荐） |
| 递归查询     | `WITH RECURSIVE`（ANSI SQL:1999）               | Oracle 的 `CONNECT BY`（早期非标准实现）                     |

#### 3. 与你熟悉的技术栈结合

- **Presto**：默认遵循 ANSI SQL 标准（例如支持标准 `JOIN`、`WITH RECURSIVE`、`OFFSET/FETCH` 分页），同时支持少量扩展语法（如 Presto 自定义的 `UNNEST` 函数）；
- **StarRocks**：2.0 版本后默认启用 ANSI SQL 模式（之前是 MySQL 兼容模式），支持标准 `JOIN`、`事务控制`、`CASE WHEN` 等，同时保留部分 MySQL 扩展（如 `LIMIT`）以兼容用户习惯；
- **MyBatis**：编写 Mapper 时，若使用 ANSI 标准 SQL，可让代码在切换数据库（如从 MySQL 迁移到 StarRocks）时无需大幅修改。

### 二、ANSI 在其他技术领域的常见含义

除了 SQL 标准，ANSI 还制定了其他与开发相关的标准：

1. **字符编码**：如 `ANSI 编码`（实际是 Windows-1252 编码，常用于早期 Windows 系统，兼容部分 ASCII 字符，但不支持 Unicode）；
2. **编程语言规范**：如 ANSI C（C 语言的 ANSI 标准，统一了 C 语言的语法和库函数，让不同编译器（GCC、Clang）编译的代码可移植）；
3. **接口 / 硬件标准**：如 ANSI 键盘布局（标准美式键盘）、ANSI 串口协议等，但这些与你关注的 Java/SQL/ 大数据领域关联较小。

### 三、开发者需要关注 ANSI 的核心原因

1. **跨平台兼容性**：使用 ANSI 标准 SQL 编写的代码，可在不同数据库（Presto、StarRocks、PostgreSQL）中直接运行，减少迁移成本；
2. **规范性**：ANSI 标准是行业通用规范，遵循标准可让代码更易读、易维护（避免依赖厂商专属语法导致的可读性下降）；
3. **功能兼容性**：新的 SQL 特性（如递归 CTE、窗口函数）通常先被 ANSI 标准化，再被各数据库厂商实现，关注 ANSI 标准可提前了解通用功能方向。

### 总结

对你的技术栈（SQL、Presto、StarRocks、MyBatis）而言，**ANSI 最核心的含义是 “ANSI SQL 标准”**—— 即 SQL 语言的统一规范，目的是实现跨数据库兼容。在实际开发中，优先使用 ANSI 标准语法（如显式 `JOIN...ON`、`WITH RECURSIVE`），可提升代码的可移植性和规范性，尤其在大数据多引擎环境（Presto 查 StarRocks、数据迁移）中更有价值。
